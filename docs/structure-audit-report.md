# Structure handler audit — templated commanders

Generated by `audit_structure.py` on a fresh scan of `internal/gameengine/per_card/gen_*.go`. Cross-references each auto-templated handler against the card's oracle text in `data/rules/ast_dataset.jsonl` and looks for effect classes the `cmd/gen-handlers` template demonstrably can't model.

## Summary

- Pool commanders (unique): **652**
- Templated handlers (`gen_*.go` files): **198**
- Templated handlers covering pool commanders: **198**
- Stub-only (template gave up, no logic emitted): **125**
- Partial (template caught the headline effect, secondary abilities unhandled): **72**
- Complete (template covered all detected effects): **1**
- No AST data available: **0**

## Methodology

1. Walk `internal/gameengine/per_card/gen_*.go` and collect every card name registered via `r.OnETB / OnCast / OnResolve / OnActivated / OnTrigger` — these names are exactly what the engine dispatches on.
2. Filter to the 652 pool commanders enumerated from `data/decks/moxfield/*.txt` `COMMANDER:` lines.
3. For each handler, classify by its `// Auto-generated <kind>` marker (`ETB`, `trigger`, `activated`, `static`) and whether the body emits `emitPartial(...)` — a generator-emitted breadcrumb that the oracle text didn't fit any of its parser branches.
4. Match the card's oracle text against a curated set of regex patterns covering effect classes the template cannot express: non-ETB triggers (whenever / upkeep / end-step), activated abilities with mana costs, replacement effects (`instead`, `as it enters`, `would`), per-opponent statics, anthems, ward, partner / choose a background / eminence, transform, dungeon / monarch / initiative / Ring, morph / manifest, level-up, companion deckbuilding, etc.
5. Cross-check whether the template DID catch any headline effect (draw / gain life / create token / scry / mill / +1/+1).

Bucketing rules:
- **Stub-only** = `static` template OR `emit_partial` with no template-handled hint. The handler is essentially a no-op.
- **Partial** = template caught the headline but oracle has additional effect classes (extra triggers, activated, replacement, etc.) that need custom code.
- **Complete** = no `emit_partial`, no detected unhandled patterns. These are usually clean ETB+draw / ETB+token cards.

## Stub-only commanders (125)

These handlers either emit nothing meaningful (`static` template, which always falls through to `emitPartial`) or hit `emitPartial` with no template-recognized headline effect. Highest priority for custom implementations — every game with these commanders is running a no-op handler.

## Caveats and known limitations

- **Activated-ability detection is conservative.** The audit only flags
  activated abilities whose cost contains a mana symbol (`{N}`, `{T}`,
  `{W}`, etc.). Tap-something or sacrifice-something costs without a
  mana symbol slip past the regex. The single "Complete" entry below
  (Azami, Lady of Scrolls — `Tap an untapped Wizard you control: Draw
  a card`) is one such case: the template's `drawOne` covers the
  effect side, but the activation cost — exiling/tapping a Wizard — is
  not enforced. Treat the "Complete" bucket as "no further effect
  classes detected by the regex set," not "this handler is provably
  correct."

- **Pattern overlap is intentionally collapsed.** When a more specific
  trigger pattern fires (`triggered_attacks`, `triggered_dies`,
  `triggered_upkeep`, `triggered_endstep`), the more general
  `triggered_non_etb` (whenever-clause) tag is dropped from the same
  row to keep the unhandled-effects column readable. Cards that fire
  on both upkeep AND a generic whenever-clause are the loss; spot
  checks suggest those are <2% of the partial bucket.

- **The 198-handler / 198-pool-coverage symmetry is not coincidence.**
  Every `gen_*.go` file's `r.OnETB / OnTrigger / OnActivated` registers
  a name that the 652-pool enumeration finds. There are zero templated
  handlers covering non-pool commanders; the generator is targeting
  the deck pool exactly. A future expansion to all 3,051 legal
  commanders would surface a much larger gen_* set and this audit
  would need to be re-run.

- **No DFC-back-face audit.** When a templated handler covers an
  MDFC commander, the generator only registers the front-face name.
  If the deck plan revolves around the back face (Esika → Bridge —
  fixed manually in PR #15), the templated handler is a no-op for
  the strategically-correct cast. The audit doesn't flag this
  separately because by the time a back-face handler exists it's
  hand-written, not in `gen_*.go`. Esika is excluded for that reason.

## Recommended next steps

1. **Stub-only `static` commanders (~80 of 125).** These are anthems,
   per-opponent effects, ward, partner/background, etc. — almost
   always continuous effects that need to thread through the layers
   system. Highest impact per fix; the engine currently runs them as
   no-ops, so any deck built around the static effect plays as if
   the commander has vanilla stats.

2. **Stub-only `ETB` commanders (~22 of 125).** Each card here has
   an ETB effect more complex than draw/gain-life/token. Examples:
   Karumonix (ETB tutor for Rats), Mabel (ETB token + grant), Tiamat
   (ETB tutor 5 dragons). One handler per card; concrete, scoped work.

3. **Stub-only `activated` commanders (~14 of 125).** These need a
   real activation handler, not just an ETB stub. Bilbo Birthday
   Celebrant's replacement effect, Captain America's shield-throw,
   etc.

4. **Partial-coverage commanders (72).** Lower priority than the
   stub bucket — the headline effect already fires. Bug-fix triage
   should run when a regression test or analytics report flags a
   specific missing behavior; otherwise leave for later batches.
